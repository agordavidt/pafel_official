================================================================================
   UNDERSTANDING APIs — A BEGINNER'S GUIDE
   Using Nomba Payment Integration as a Real-World Case Study
   Written for absolute beginners. No prior coding knowledge assumed.
================================================================================

Hello and welcome! By the end of this lesson, you will understand what APIs
are, how they work, and how real companies like Nomba use them to power
payments. We will also walk through building a simple payment page step by
step. Take your time — every concept is explained with everyday examples first.

--------------------------------------------------------------------------------
LESSON 1: WHAT IS AN API?
--------------------------------------------------------------------------------

Let's start with a story.

Imagine you walk into a restaurant. You sit down, look at the menu, and when
you are ready, a waiter comes to take your order. You tell the waiter: "I want
jollof rice and a bottle of water." The waiter walks to the kitchen, tells the
chef what you want, the chef prepares it, and the waiter brings it back to you.

You never went into the kitchen yourself. You never talked to the chef directly.
The WAITER was the middleman — and that is exactly what an API is.

API stands for Application Programming Interface.

In simple terms:
- YOU are the app or website that needs something (e.g., "process this payment")
- The KITCHEN is another system that has what you need (e.g., Nomba's servers)
- The WAITER (API) is the messenger that carries your request and brings back
  the response

So when your app wants to charge a customer ₦2,500, it "calls" Nomba's API,
which talks to the bank, processes the payment, and brings back a result:
"Payment successful!" or "Payment failed."

WHY DO WE NEED APIS?

You don't need to build a bank to accept payments. Nomba already has all the
banking infrastructure. They expose their system through an API so that any
developer can plug in and use it — just like how you plug your phone charger
into a socket. You did not build the electricity grid, but you can use it.

APIs allow developers to:
1. Save time — reuse what others have already built
2. Focus on their product — let specialists handle payments, maps, messaging
3. Connect different systems — your Laravel app can talk to Nomba, Google,
   WhatsApp, and more


--------------------------------------------------------------------------------
LESSON 2: HOW DO APPS TALK TO EACH OTHER? (HTTP & REQUESTS)
--------------------------------------------------------------------------------

When you open a website, your browser sends a REQUEST to a server, and the
server sends back a RESPONSE. APIs work exactly the same way.

There are a few types of requests (called HTTP methods). The most common are:

   POST  —  "I am sending you something, please do something with it."
            Example: Sending payment details to Nomba to create an order.

   GET   —  "Please give me some information."
            Example: Asking Nomba for the status of a payment.

   DELETE — "Please remove this."
            Example: Deleting a saved card token.

Every API has an ADDRESS called a URL (like a website address). For Nomba:

   Sandbox (for testing): https://sandbox.api.nomba.com/v1
   Production (real money): https://api.nomba.com/v1

Think of the Sandbox as a training ground — it behaves exactly like the real
system but no real money moves. You can make "pretend" payments to test your
code without losing a single kobo. This is very important: ALWAYS test in
sandbox first.

WHAT IS A RESPONSE?

When you send a request, the API responds with data, usually in a format called
JSON (JavaScript Object Notation). Don't be scared by the name — JSON is just
structured text that both humans and computers can read easily.

Example JSON response from Nomba after creating an order:

   {
     "data": {
       "checkoutUrl": "https://checkout.nomba.com/pay/abc123",
       "orderReference": "ord_12345"
     }
   }

This is just like the waiter coming back and saying: "Here is your table number
and the link to track your order." Your app reads this JSON and uses the
checkoutUrl to redirect your customer to the payment page.


--------------------------------------------------------------------------------
LESSON 3: AUTHENTICATION — PROVING WHO YOU ARE
--------------------------------------------------------------------------------

Imagine you call a bank and say "Please send ₦50,000 from Account 12345." The
bank won't just do it. They will ask: "Who are you? What is your password?"

APIs work the same way. Before Nomba will process any payment for you, you must
PROVE that you are a registered merchant with a valid account. This process is
called AUTHENTICATION.

Nomba uses a system called OAuth 2.0. Again, don't worry about the name. Here
is how it works in plain English:

STEP 1: You introduce yourself
You send Nomba your clientId (your username) and your clientSecret (your
password). Think of these as the keys to your shop.

   VERY IMPORTANT: Never share your clientSecret with anyone. Never put it
   directly in your code where others can see it. Store it in a secret place
   called an environment variable (we will explain this shortly).

STEP 2: Nomba gives you a temporary badge
If your credentials are correct, Nomba gives you an ACCESS TOKEN — a temporary
pass that lasts for 60 minutes. It looks like a long string of random characters.

   Example: "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..."

STEP 3: You show your badge on every request
Every time you ask Nomba to do something, you include this access token as proof
that you are allowed. This is called a Bearer Token.

   You attach it to every request like this:
   Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

WHAT IS AN ENVIRONMENT VARIABLE?

An environment variable is like a secret note stored on your computer or server
that your code can read, but it is NOT written inside your code. If someone
looks at your source code, they will not see your secret.

Instead of writing this in your code (WRONG — never do this):
   client_secret = "my_super_secret_key_12345"

You store it in a file called .env:
   NOMBA_CLIENT_SECRET=my_super_secret_key_12345

And your code reads it like this:
   secret = os.environ.get("NOMBA_CLIENT_SECRET")

Think of it like this: your code says "the password is in the safe," but does
not actually write the password on the wall.

TOKEN CACHING (Being Smart About Tokens)

Remember, tokens expire in 60 minutes. It would be wasteful to ask for a new
token every single time your code runs. Instead, you get a token, save it
temporarily (in memory or a cache like Redis), and reuse it. At the 55-minute
mark, you refresh it before it expires.

It is like a shop pass that lasts a day — you don't go back to the gate every
time you move between floors. You got it once, use it all day, renew it before
it expires.


--------------------------------------------------------------------------------
LESSON 4: ONLINE CHECKOUT — COLLECTING PAYMENTS
--------------------------------------------------------------------------------

Now to the exciting part — collecting money!

The Checkout API is the simplest way to collect payment. Instead of building
your own payment form (which is very complicated and a security nightmare),
Nomba provides a HOSTED PAYMENT PAGE — a ready-made, secure page where your
customer enters their card details.

Here is how it works:

1. Your app sends an order to Nomba: "I want to charge ₦2,500 to customer Ada."
2. Nomba creates a checkout session and gives you a special URL (checkoutUrl).
3. You redirect Ada's browser to that URL.
4. Ada sees a beautiful payment form, enters her card details.
5. Nomba handles everything — security checks, OTP confirmation, 3D Secure.
6. When payment is done, Nomba redirects Ada back to YOUR website (callbackUrl).
7. Nomba also sends a message to your server in the background (webhook).

THE MOST IMPORTANT THING — AMOUNTS ARE IN KOBO!

This catches everyone the first time. In Nigeria, we think in Naira, but the
Nomba API uses KOBO.

   ₦1.00  =  100 kobo
   ₦25.00 =  2,500 kobo
   ₦2,500 =  250,000 kobo

So if you want to charge ₦2,500, you must send the number 250000 to the API.
If you send 2500, you will charge just ₦25. Always multiply by 100!

A REAL EXAMPLE:

You send this to Nomba:
   {
     "order": {
       "orderReference": "ord_001",
       "amount": 250000,
       "currency": "NGN",
       "callbackUrl": "https://yourshop.com/payment/done",
       "customerEmail": "ada@gmail.com"
     }
   }

Nomba responds with:
   {
     "data": {
       "checkoutUrl": "https://checkout.nomba.com/pay/xyz789"
     }
   }

Your code then redirects Ada to that URL. She pays, comes back to your site.
Done!


--------------------------------------------------------------------------------
LESSON 5: WEBHOOKS — NOMBA CALLS YOUR SERVER
--------------------------------------------------------------------------------

Here is a question: What if Ada pays but her internet disconnects before Nomba
redirects her back to your site? Did the payment go through? How do you know?

This is where WEBHOOKS come in.

A webhook is like Nomba setting a reminder to call your phone the moment
something happens — regardless of whether the customer is still on the page.

In technical terms: a webhook is a URL on YOUR server that Nomba will send a
POST request to when an event occurs. Events include:
- Payment successful
- Payment failed
- Virtual account was funded
- Transfer completed

You tell Nomba: "My webhook URL is https://yourshop.com/webhooks/nomba."
Whenever a payment happens, Nomba will send the details to that URL.

WEBHOOK SECURITY — SIGNATURES

Now, what if a bad person figures out your webhook URL and sends you fake
"payment successful" messages? Your server would think payments happened when
they didn't!

To prevent this, Nomba SIGNS every webhook. It uses your webhook secret to
create a unique fingerprint of the message (called an HMAC-SHA256 signature).

When your server receives a webhook, you:
1. Take the raw message
2. Use your webhook secret to calculate what the fingerprint SHOULD be
3. Compare it to the fingerprint Nomba sent
4. If they match — it is genuinely from Nomba. If not — reject it.

This is like a wax seal on a letter. Only someone with the original stamp could
have sealed it. If the seal is broken or wrong, you know the letter was tampered
with.

IDEMPOTENCY — DON'T DO THINGS TWICE

What if Nomba's webhook fires twice for the same payment? (This can happen due
to network issues.) You don't want to mark an order as paid twice, or ship
something twice.

The solution: every webhook has a unique requestId. Before processing a webhook,
check if you have already processed that requestId. If yes, ignore it. This
property — being safe to receive the same message multiple times — is called
IDEMPOTENCY.


--------------------------------------------------------------------------------
LESSON 6: VIRTUAL ACCOUNTS — BANK TRANSFER PAYMENTS
--------------------------------------------------------------------------------

Not everyone has a card. Many Nigerians prefer to pay by bank transfer. Virtual
accounts solve this.

A virtual account is a REAL bank account number (NUBAN format) that Nomba
creates specifically for a transaction or a customer. You give this account
number to your customer. They transfer money to it from any Nigerian bank.
The moment the transfer lands, Nomba fires a webhook to your server: "Account
inv_9921 just received ₦10,000 from GTBank."

This is perfect for:
- Invoicing: "Please pay ₦10,000 to account 0123456789 at Nomba Bank"
- Marketplaces: Give each seller their own dedicated account number
- Escrow: Hold money in a virtual account until conditions are met

HANDLING OVER AND UNDER-PAYMENTS

Banks don't stop people from sending the wrong amount. If an invoice is for
₦10,000 and someone sends ₦9,500, the money still arrives. Your code must
compare what was expected to what was actually received and handle both cases:
- Underpayment: Alert the customer that they are short
- Overpayment: Refund the extra


--------------------------------------------------------------------------------
LESSON 7: TOKENIZED CARDS — SAVING CARDS FOR LATER
--------------------------------------------------------------------------------

Have you ever used an app where you pay once and the second time it just
charges you without asking for your card again? That is card tokenization.

After a customer pays through Nomba's checkout, Nomba can give you back a
CARD TOKEN — a safe reference to that card. It is not the actual card number
(that would be dangerous), just a code like "tok_5fa12b..."

You save this token in your database linked to the customer. The next time they
want to pay, you charge the token — no card details needed.

This is how subscription services work:
1. Month 1: Customer pays through checkout, you get a token
2. Month 2: Your system automatically charges the token (no customer action)
3. Month 3: Repeat

IMPORTANT: Nomba does not run the subscription schedule for you. YOU must build
the scheduler (a cron job) that runs every month and charges the token.


--------------------------------------------------------------------------------
LESSON 8: SUB-ACCOUNTS — SPLITTING MONEY BETWEEN SELLERS
--------------------------------------------------------------------------------

Imagine you are building an online marketplace — like Jumia or Konga. You have
many sellers, and when a customer buys something, you need to track whose money
is whose, and eventually pay each seller their share.

Sub-accounts let you create a separate "mini-wallet" inside your Nomba merchant
account for each seller. Each sub-account has its own balance and its own
virtual account. You can move money between them and pay out sellers
individually.

Example:
- Main account: YourMarketplace (parent)
  - Sub-account: Adaeze Kitchen (seller 1)
  - Sub-account: Emeka Phones (seller 2)
  - Sub-account: Ibrahim Fashion (seller 3)

When a customer buys from Adaeze, you put the money in Adaeze's sub-account.
When you pay her out, you transfer from her sub-account to her bank.


--------------------------------------------------------------------------------
LESSON 9: TRANSFERS — PAYING MONEY OUT
--------------------------------------------------------------------------------

While the previous APIs were about collecting money IN, transfers are about
sending money OUT — to sellers, for refunds, for payouts.

Before you transfer, ALWAYS verify the recipient's account number using the
bank lookup API. This resolves the account number to a name so you can confirm
you are sending to the right person. Do this before every transfer — even if you
think you know the account.


--------------------------------------------------------------------------------
LESSON 10: RECONCILIATION — MAKING SURE EVERYTHING MATCHES
--------------------------------------------------------------------------------

Reconciliation means comparing two lists to make sure they match.

Your app has a record of payments. Nomba has a record of payments. Every night,
you should pull Nomba's transaction list and compare it to your own records. If
there is any difference — money you received that you did not record, or money
you recorded that Nomba has no record of — you MUST investigate.

This is the equivalent of a cashier counting the till at the end of the day and
comparing it to the POS receipts. It is not optional. Skipping reconciliation
is how businesses silently lose money.

Nomba provides a /transactions endpoint where you can filter by date, status,
and type to pull all transactions for any period.

STEP 4: CREATE app.py (YOUR BACKEND)
--------------------------------------------------------------------------------

Open app.py and type the following (every line is explained with comments):

---------- app.py ----------

import os
import uuid
import requests
from flask import Flask, render_template, redirect, request
from dotenv import load_dotenv

# Load secrets from .env file
load_dotenv()

app = Flask(__name__)

# Read credentials from environment variables (NOT hardcoded!)
BASE_URL     = os.environ.get("NOMBA_BASE_URL")
ACCOUNT_ID   = os.environ.get("NOMBA_ACCOUNT_ID")
CLIENT_ID    = os.environ.get("NOMBA_CLIENT_ID")
CLIENT_SECRET = os.environ.get("NOMBA_CLIENT_SECRET")


def get_access_token():
    """
    Step 1: Authenticate with Nomba.
    We send our clientId and clientSecret.
    Nomba gives us back a temporary access token (valid 60 minutes).
    Think of this as logging in to get a session badge.
    """
    url = f"{BASE_URL}/auth/token/issue"

    headers = {
        "Content-Type": "application/json",
        "accountId": ACCOUNT_ID,
    }

    body = {
        "grant_type": "client_credentials",
        "client_id": CLIENT_ID,
        "client_secret": CLIENT_SECRET,
    }

    response = requests.post(url, json=body, headers=headers)
    data = response.json()

    # The token is inside data.data.access_token
    return data["data"]["access_token"]


def create_checkout(token, amount_naira, customer_email):
    """
    Step 2: Create a checkout session.
    We tell Nomba: "I want to charge this customer this amount."
    Nomba gives us a checkoutUrl — the hosted payment page link.
    
    amount_naira: the amount in Naira (we will convert to kobo inside)
    """
    url = f"{BASE_URL}/checkout/order"

    # Convert Naira to Kobo (multiply by 100)
    amount_kobo = amount_naira * 100

    # Generate a unique order reference — every order must have a unique ID
    order_ref = "ord_" + str(uuid.uuid4())

    headers = {
        "Content-Type": "application/json",
        "Authorization": f"Bearer {token}",  # <-- attach our badge
        "accountId": ACCOUNT_ID,
    }

    body = {
        "order": {
            "orderReference": order_ref,
            "amount": amount_kobo,
            "currency": "NGN",
            "customerEmail": customer_email,
            # After payment, Nomba will send the user back to this URL
            "callbackUrl": "http://localhost:5000/success",
        }
    }

    response = requests.post(url, json=body, headers=headers)
    data = response.json()

    return data["data"]["checkoutUrl"]


# ---- ROUTES ----

@app.route("/")
def index():
    """Show the homepage with a payment button."""
    return render_template("index.html")


@app.route("/pay")
def pay():
    """
    When the user clicks "Pay", this runs.
    1. Get an access token from Nomba
    2. Create a checkout session
    3. Redirect the user to Nomba's payment page
    """
    try:
        # Step 1: Login to Nomba
        token = get_access_token()

        # Step 2: Create the payment session
        # Charging ₦2,500 for a demo product
        checkout_url = create_checkout(
            token=token,
            amount_naira=2500,
            customer_email="customer@example.com"
        )

        # Step 3: Send the user to Nomba's payment page
        return redirect(checkout_url)

    except Exception as e:
        return f"Something went wrong: {str(e)}", 500


@app.route("/success")
def success():
    """
    Nomba redirects the user back here after payment.
    In a real app, you would verify the payment status using the
    orderReference before marking the order as paid.
    """
    return render_template("success.html")


if __name__ == "__main__":
    app.run(debug=True)

---------- end of app.py ----------

================================================================================
   KEY LESSONS SUMMARY
================================================================================

Concept              What it means (in plain English)
-------------------  ---------------------------------------------------------
API                  A waiter that carries messages between two systems
HTTP Methods         GET = ask for data; POST = send data; DELETE = remove data
Base URL             The address of the API (like a phone number)
Sandbox              A safe practice environment — no real money involved
Authentication       Proving who you are before the API trusts you
clientId/Secret      Your username and password for the API
Access Token         A temporary badge (60 min) that authorises your requests
Bearer Token         How you attach your badge to every request
Kobo                 Nomba uses kobo not naira. ₦1 = 100 kobo. Always × 100
Checkout URL         A Nomba-hosted payment page — no card handling on your end
Callback URL         Where Nomba sends the user BACK to after payment
Webhook              Nomba calling YOUR server to report what happened
HMAC Signature       A security stamp that proves a webhook is genuinely Nomba's
Idempotency          Processing a message only once, even if it arrives twice
Virtual Account      A real NUBAN account Nomba creates for you to receive transfers
Card Token           A safe reference to a saved card for future charges
Sub-account          A mini-wallet inside your Nomba account (e.g., per seller)
Reconciliation       Nightly check: does YOUR record match NOMBA's record?
Environment Variable A secret stored outside your code (use a .env file)


================================================================================
   NEXT STEPS — WHERE TO GO FROM HERE
================================================================================

1. ADD WEBHOOK HANDLING
   Build a /webhooks/nomba route in your Flask app that receives Nomba's
   POST requests. Verify the HMAC signature and mark orders as paid based
   on the webhook, not just the redirect. This is the production-correct way.

2. STORE PAYMENT RECORDS IN A DATABASE
   Use SQLite (beginner-friendly) or MySQL to save every order, its
   orderReference, status, and amount. Update the status when you receive
   the webhook.

3. VERIFY PAYMENT BEFORE CONFIRMING ORDER
   After the user returns to /success, call GET /checkout/order/{orderReference}
   to confirm the payment status before showing "order confirmed."

4. ADD TOKEN CACHING
   Instead of calling /auth/token/issue on every request, save the token
   in memory with its expiry time. Reuse it until 5 minutes before expiry.

5. EXPLORE VIRTUAL ACCOUNTS
   Add a "Pay by Transfer" option that creates a virtual account and shows
   the customer a bank account number to transfer to.

6. MOVE TO PHP/LARAVEL (your stack)
   Everything we did here maps directly to Laravel. Use Guzzle (Laravel's
   HTTP client) instead of Python's requests library. The logic is identical.


================================================================================
   FINAL WORDS
================================================================================

APIs are not magic — they are just structured conversations between systems.
Once you understand that every API call is just:
   1. Prove who you are (authentication)
   2. Ask for something or send something (request)
   3. Read the response (JSON)
   4. React to what happened (webhooks, callbacks)

...you can integrate almost any API in the world. Nomba's documentation follows
the same patterns as Stripe, Paystack, Flutterwave, and every other payment
gateway.

The skills you learned today transfer everywhere.

Good luck building! 🚀
